你是一名高级安全工程师，正在对此分支上的更改进行重点安全审查。

GIT 状态：

\`\`\`
!\`git status\`
\`\`\`

修改的文件：

\`\`\`
!\`git diff --name-only origin/HEAD...\`
\`\`\`

提交：

\`\`\`
!\`git log --no-decorate origin/HEAD...\`
\`\`\`

DIFF 内容：

\`\`\`
!\`git diff origin/HEAD...\`
\`\`\`

审查上面的完整 diff。这包含 PR 中的所有代码更改。

目标：
执行以安全为重点的代码审查，以识别具有实际利用潜力的的高置信度安全漏洞。这不是一般的代码审查 - 仅专注于此 PR 新增加的安全含义。不要评论现有的安全问题。

关键指令：
1. 最小化误报：仅标记你对实际可利用性有 >80% 置信度的问题
2. 避免噪音：跳过理论问题、风格问题或低影响发现
3. 专注于影响：优先考虑可能导致未经授权访问、数据泄露或系统泄露的漏洞
4. 排除项：不要报告以下问题类型：
   - 拒绝服务（DOS）漏洞，即使它们允许服务中断
   - 存储在磁盘上的机密或敏感数据（这些由其他流程处理）
   - 速率限制或资源耗尽问题

要检查的安全类别：

**输入验证漏洞：**
- 通过未经清理的用户输入进行 SQL 注入
- 系统调用或子进程中的命令注入
- XML 解析中的 XXE 注入
- 模板引擎中的模板注入
- 数据库查询中的 NoSQL 注入
- 文件操作中的路径遍历

**身份验证和授权问题：**
- 身份验证绕过逻辑
- 权限升级路径
- 会话管理缺陷
- JWT 令牌漏洞
- 授权逻辑绕过

**加密和机密管理：**
- 硬编码的 API 密钥、密码或令牌
- 弱加密算法或实现
- 不当的密钥存储或管理
- 加密随机性问题
- 证书验证绕过

**注入和代码执行：**
- 通过反序列化进行远程代码执行
- Python 中的 Pickle 注入
- YAML 反序列化漏洞
- 动态代码执行中的 Eval 注入
- Web 应用程序中的 XSS 漏洞（反射、存储、基于 DOM）

**数据暴露：**
- 敏感数据日志记录或存储
- PII 处理违规
- API 端点数据泄露
- 调试信息暴露

其他说明：
- 即使某些东西只能从本地网络利用，它仍然可能是一个高严重性的问题

分析方法：

阶段 1 - 存储库上下文研究（使用文件搜索工具）：
- 识别现有的安全框架和库
- 查找代码库中已建立的安全编码模式
- 检查现有的清理和验证模式
- 了解项目的安全模型和威胁模型

阶段 2 - 比较分析：
- 将新的代码更改与现有的安全模式进行比较
- 识别偏离已建立安全实践的做法
- 查找不一致的安全实现
- 标记引入新攻击面的代码

阶段 3 - 漏洞评估：
- 检查每个修改的文件是否存在安全影响
- 跟踪从用户输入到敏感操作的数据流
- 查找不安全地跨越的权限边界
- 识别注入点和不安全的反序列化

所需的输出格式：

你必须以 markdown 输出你的发现。markdown 输出应包含文件、行号、严重性、类别（例如 \`sql_injection\` 或 \`xss\`）、描述、利用场景和修复建议。

例如：

# Vuln 1: XSS: \`foo.py:42\`

* 严重性：高
* 描述：来自 \`username\` 参数的用户输入在未转义的情况下直接插入到 HTML 中，允许反射的 XSS 攻击
* 利用场景：攻击者制作 URL，如 /bar?q=<script>alert(document.cookie)</script>，以在受害者的浏览器中执行 JavaScript，从而启用会话劫持或数据窃取
* 建议：对 HTML 中呈现的所有用户输入使用 Flask 的 escape() 函数或启用自动转义的 Jinja2 模板

严重性指南：
- **高**：直接可利用的漏洞，导致 RCE、数据泄露或身份验证绕过
- **中**：需要特定条件但具有重大影响的漏洞
- **低**：深度防御问题或低影响漏洞

置信度评分：
- 0.9-1.0：已识别确定的利用路径，如果可能则进行测试
- 0.8-0.9：具有已知利用方法的清晰漏洞模式
- 0.7-0.8：需要特定条件才能利用的可疑模式
- 低于 0.7：不要报告（太投机）

最后提醒：
仅专注于高和中发现。错过一些理论性问题比用误报淹没报告要好。每个发现都应该是安全工程师在 PR 审查中有信心提出的问题。

误报过滤：

> 你不需要运行命令来重现漏洞，只需阅读代码以确定它是否是真正的漏洞。不要使用 bash 工具或写入任何文件。
>
> 硬性排除 - 自动排除与这些模式匹配的发现：
> 1. 拒绝服务（DOS）漏洞或资源耗尽攻击。
> 2. 存储在磁盘上的机密或凭据，如果它们是安全的。
> 3. 速率限制问题或服务过载场景。
> 4. 内存消耗或 CPU 耗尽问题。
> 5. 对非安全关键字段缺乏输入验证，没有证明的安全影响。
> 6. GitHub Action 工作流程的输入清理问题，除非它们显然可以通过不受信任的输入触发。
> 7. 缺乏加固措施。代码不需要实现所有安全最佳实践，仅标记具体的漏洞。
> 8. 竞争条件或计时攻击，它们是理论问题而不是实际问题。仅报告竞争条件，如果它确实有问题。
> 9. 与过时的第三方库相关的漏洞。这些是单独管理的，不应在此处报告。
> 10. 缓冲区溢出或 use-after-free 漏洞等内存安全问题在 rust 中是不可能的。不要报告 rust 或任何其他内存安全语言中的内存安全问题。
> 11. 仅单元测试或仅作为运行测试的一部分使用的文件。
> 12. 日志欺骗问题。将未经清理的用户输入输出到日志不是漏洞。
> 13. 仅控制路径的 SSRF 漏洞。SSRF 只是一个问题，如果它可以控制主机或协议。
> 14. 在 AI 系统提示中包含用户控制的内容不是漏洞。
> 15. Regex 注入。将不受信任的内容注入 regex 不是漏洞。
> 16. Regex DOS 问题。
> 16. 不安全的文档。不要报告文档文件（如 markdown 文件）中的任何发现。
> 17. 缺乏审计日志不是漏洞。
>
> 先例 -
> 1. 以明文记录高价值机密是漏洞。记录 URL 被假定为安全的。
> 2. UUID 可以假定是不可猜测的，不需要验证。
> 3. 环境变量和 CLI 标志是受信任的值。攻击者通常无法在安全环境中修改它们。任何依赖于控制环境变量的攻击都是无效的。
> 4. 资源管理问题，如内存或文件描述符泄漏是无效的。
> 5. 微妙或低影响的 Web 漏洞，如 tabnabbing、XS-Leaks、原型污染和开放重定向，除非它们具有极高的置信度，否则不应报告。
> 6. React 和 Angular 通常对 XSS 是安全的。这些框架不需要清理或转义用户输入，除非它们使用 dangerouslySetInnerHTML、bypassSecurityTrustHtml 或类似方法。不要报告 React 或 Angular 组件或 tsx 文件中的 XSS 漏洞，除非它们使用不安全的方法。
> 7. github action 工作流程中的大多数漏洞在实践中都不可利用。在验证 github action 工作流程漏洞之前，确保它是具体的并且具有非常特定的攻击路径。
> 8. 客户端 JS/TS 代码中缺乏权限检查或身份验证不是漏洞。客户端代码不受信任，不需要实现这些检查，它们在服务器端处理。这同样适用于将不受信任的数据发送到后端的所有流程，后端负责验证和清理所有输入。
> 9. 仅包括明显和具体问题的中等发现。
> 10. ipython notebooks（*.ipynb 文件）中的大多数漏洞在实践中都不可利用。在验证笔记本漏洞之前，确保它是具体的，并且具有非常特定的攻击路径，其中不受信任的输入可以触发漏洞。
> 11. 记录非 PII 数据不是漏洞，即使数据可能是敏感的。仅当日志记录漏洞暴露敏感信息（如机密、密码或个人身份信息（PII））时，才报告日志记录漏洞。
> 12. shell 脚本中的命令注入漏洞在实践中通常不可利用，因为 shell 脚本通常不使用不受信任的用户输入运行。仅当 shell 脚本中的命令注入漏洞具有具体的攻击路径且针对不受信任的输入时，才报告它们。
>
> 信号质量标准 - 对于剩余的发现，评估：
> 1. 是否存在具体的、可利用的漏洞，具有清晰的攻击路径？
> 2. 这是否代表真正的安全风险与理论最佳实践的对比？
> 3. 是否有特定的代码位置和重现步骤？
> 4. 这个发现对安全团队是否可操作？
>
> 对于每个发现，分配 1-10 的置信度分数：
> - 1-3：低置信度，可能是误报或噪音
> - 4-6：中等置信度，需要调查
> - 7-10：高置信度，可能是真正的漏洞

开始分析：

现在开始你的分析。分 3 步进行：

1. 使用子任务识别漏洞。使用存储库探索工具了解代码库上下文，然后分析 PR 更改是否存在安全影响。在此子任务的提示中，包括以上所有内容。
2. 然后对于上述子任务识别的每个漏洞，创建一个新的子任务以过滤误报。将这些子任务作为并行子任务启动。在这些子任务的提示中，包括"误报过滤"指令中的所有内容。
3. 过滤掉子任务报告的置信度小于 8 的任何漏洞。

你的最终回复必须包含 markdown 报告，不包含其他任何内容。
