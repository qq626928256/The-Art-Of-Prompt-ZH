你的任务是创建迄今为止对话的详细摘要，密切关注用户的明确请求和你之前的操作。
此摘要应彻底捕捉技术细节、代码模式和架构决策，这些对于继续开发工作而不丢失上下文至关重要。

${ANALYSIS_INSTRUCTION_TAGS}

你的摘要应包括以下部分：

1. 主要请求和意图：详细捕获所有用户的明确请求和意图
2. 关键技术概念：列出所有讨论的重要技术概念、技术和框架。
3. 文件和代码部分：枚举检查、修改或创建的特定文件和代码部分。特别关注最近的邮件，并在适用时包括完整的代码片段，包括关于为什么此文件读取或编辑很重要的摘要。
4. 错误和修复：列出你遇到的所有错误以及你如何修复它们。特别注意你收到的具体用户反馈，特别是如果用户告诉你以不同方式做某事。
5. 问题解决：记录已解决的问题和任何正在进行的故障排除工作。
6. 所有用户消息：列出所有非工具结果的用户消息。这些对于理解用户反馈和变化的意图至关重要。
7. 待处理任务：概述你明确被要求处理的任何待处理任务。
8. 当前工作：详细描述在此摘要请求之前立即进行的工作，特别注意来自用户和 assistant 的最近消息。在适用时包括文件名和代码片段。
9. 可选的下一步：列出你将采取的与最近正在进行的工作相关的下一步。重要提示：确保这一步直接与用户最近的明确请求以及你在此次摘要请求之前立即进行的任务保持一致。如果你的最后一个任务已结束，则仅列出与用户请求明确一致的后续步骤。在首先与用户确认之前，不要开始与相关的请求或已经完成的非常旧的请求。
                       如果有下一步，请包括最近对话中的直接引用，准确显示你正在处理的任务以及你停止的地方。这应该是逐字的，以确保任务解释没有漂移。

以下是你的输出应如何结构的示例：

<example>
<analysis>
[你的思维过程，确保所有点都被彻底和准确地涵盖]
</analysis>

<summary>

2. Key Technical Concepts:
   - [概念 1]
   - [概念 2]
   - [...]

3. Files and Code Sections:
   - [文件名 1]
      - [关于此文件为何重要的摘要]
      - [对此文件所做的更改的摘要（如果有）]
      - [重要代码片段]
   - [文件名 2]
      - [重要代码片段]
   - [...]

4. Errors and fixes:
    - [错误 1 的详细描述]:
      - [你如何修复错误]

    - [...]

    - [详细的非工具使用用户消息]
    - [...]

   - [任务 1]

   - [...]

   [当前工作的精确描述]

   [要采取的可选下一步]

</summary>
</example>

请根据迄今为止的对话提供你的摘要，遵循此结构并确保响应的精确性和彻底性。

可能包含在所包含上下文中的其他摘要指令。如果是这样，请记住在创建上述摘要时遵循这些指令。指令示例包括：
<example>

摘要对话时专注于 typescript 代码更改，还要记住你犯的错误以及你如何修复它们。
</example>
